# Agent Roles

This section gives workshop attendees a practical way to use AI as a small startup team instead of a single all-purpose assistant.

Each role has a focused job:
- **Market Researcher** — finds adjacent products, patterns, competitors, and framing
- **Idea Validator** — pressure-tests assumptions, value, and MVP scope
- **Coder** — scaffolds, builds, fixes, and iterates on the app
- **Project Manager** — breaks work into milestones and sequences execution
- **Outreach Assistant** — writes positioning, launch copy, recap language, and follow-up messaging

## Why These Roles Exist
A lot of people use AI like a magic 8-ball with autocomplete. That works for random tasks, but it falls apart when you're trying to build something coherent.

These role files are designed to show a better pattern: give the AI a job, a boundary, and a clear output shape.

That makes the work:
- more consistent
- easier to steer
- easier to evaluate
- easier to combine into a real workflow

## Recommended Workshop Flow
If you're running through the OpenClaw Dev Days startup exercise, use the roles in roughly this order:

1. **Market Researcher**
   - identify adjacent products and patterns
   - find what feels familiar versus differentiated

2. **Idea Validator**
   - challenge the value proposition
   - cut weak scope and tighten the MVP

3. **Project Manager**
   - turn the idea into milestones and task order
   - define the shortest path to a usable demo

4. **Coder**
   - build the MVP and iterate based on feedback

5. **Outreach Assistant**
   - explain what was built
   - create launch copy, demo language, and follow-up messaging

## How To Use These Files
Each role folder contains an `AGENTS.md` file you can use as a starting instruction set for your AI coding or chat workflow.

You can:
- copy the file into your prompt context
- adapt it for Copilot, Codex, Claude Code, or another agent workflow
- use one role at a time
- combine them as a multi-agent team

## Suggested Teaching Pattern
A strong workshop pattern is:
- start with one shared app idea
- assign different roles to different attendees or tabs
- compare outputs
- merge the best recommendations into one build plan
- ship a working MVP before polishing the edges
