# OpenClaw Dev Days Curriculum Improvement Brief

## Purpose

This brief captures public-safe curriculum improvements for OpenClaw Dev Days.

It is intentionally scoped to reusable workshop design, attendee experience, setup flow, Markdown memory, agent roles, and follow-up learning.

## Curriculum priorities

1. Get attendees to one working OpenClaw loop quickly.
2. Teach Markdown as the shared thinking layer for humans, agents, GitHub, and Obsidian.
3. Use a small number of clear screenshots in the first hour instead of a live Markdown demo.
4. Install prerequisites together, including Obsidian.
5. Keep the first Markdown artifact in the attendee's OpenClaw workspace.
6. Include a public-repo safety checkpoint before GitHub activity.
7. Use Markdown task lists as the bridge from notes into agent work plans.
8. Make the Beaver Badges README improvement task an early lightweight agent-team exercise.
9. Reference GitHub's Markdown learning material as optional follow-up.
10. Keep workshop content reusable for future community events.

## Recommended first-hour flow

- Explain what OpenClaw is and what it is not.
- Explain agents in practical terms: role, context, tools, memory, output, review.
- Explain that models use context, not human memory.
- Show Markdown as structured plain text that helps people and agents understand the same information.
- Show screenshots of Markdown source, rendered Markdown, OpenClaw context files, agent instructions, task lists, Obsidian, and GitHub rendering.
- Reinforce that the human remains the orchestrator.

## Public safety checkpoint

Before attendees publish anything to GitHub or a shared workspace, say clearly:

> Do not put secrets, API keys, private data, client information, sensitive personal notes, or anything confidential in a public repository or shared note. When in doubt, keep it local.
