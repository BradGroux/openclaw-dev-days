Core Components

The architecture is easier to teach when each part has a job title.

Gateway

Always-on switchboard

Owns routing, channels, WebSocket protocol, pairing, health, presence, heartbeat, cron, and agent stream events.

Runtime

Turns intent into work

Assembles context, resolves models, runs the agent loop, calls tools, and returns streamed output.

Context

What the model sees

Includes system prompt, workspace files, history, tool schemas, results, attachments, and compaction summaries.

Nodes

Local capabilities

Expose device or host commands such as canvas, camera, screen, location, browser, or headless work.

Request Flow

A practical trace from a person asking for work to OpenClaw returning a durable result.

Input arrives

A client, channel, CLI command, heartbeat, or automation sends work into the Gateway.

Gateway routes

The Gateway validates the request, resolves the surface, and emits or accepts the agent run.

Context assembles

Workspace files, skills, session history, tool schemas, and runtime settings shape the prompt.

Model and tools act

The model reasons, calls tools, uses nodes when needed, and streams assistant and tool deltas.

State persists

The transcript, metadata, memory, and final response remain available for future runs.

Agent Loop

The loop is the real execution path, not a decorative chat wrapper.

What happens inside a run

  • Resolve the session key, workspace, model, auth profile, and run settings.
  • Load relevant skills and bootstrap files into the prompt-building path.
  • Serialize runs per session so tool work and transcript writes stay consistent.
  • Bridge assistant, tool, and lifecycle streams back through Gateway events.

What facilitators should say

  • Gateway is where work enters and leaves.
  • Runtime is where context, models, tools, and state come together.
  • Markdown files are durable context, not just notes.
  • Verification belongs in the workflow before real-world automation expands.

Wire Protocol

The teaching version: connect first, request with ids, stream events back.

Connect

{
  "type": "connect",
  "params": {
    "role": "client"
  }
}

Request

{
  "type": "req",
  "id": "run-101",
  "method": "agent",
  "params": { }
}

Event

{
  "type": "event",
  "event": "agent",
  "payload": {
    "stream": "tool"
  }
}

Operating Guardrails

Keep the first win local, visible, and easy to verify.

Local first

Loopback before remote

Prove the Gateway, dashboard, and first response before adding remote access or messaging channels.

Pairing

Trust devices deliberately

New clients and nodes should have clear identity and approval expectations.

Secrets

Keep private data out

Never put keys, client data, credentials, or sensitive notes in public workshop materials.

Verify

Inspect before scaling

Use health checks, status, context inspection, and human approval before risky automation.
